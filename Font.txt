 ,------------.
(  BBC MODE-7  )
 `------------'

The Acorn BBC Micro "Mode-7" is actually a special use-case of
	Mullard SAA-5050 Teletext Character Generator [505x - English (UK) Variant]

The Acorn BBC Micro has some notable anomalies.
	# Character set is presented as two 256 character fonts
	# A few strange character re-mappings
	# Additional Control Codes

This document will document the 505x ICs, and comment on deviations in "Mode-7".


 ,------------------.
(  MULLARD SAA-505x  )
 `------------------'

By current [2022] ways of thinking, a 505x IC might be thought of as presenting
"two fonts", but it is a /single/ VDU "Character Set", which all (commonly)
have/had an "Alphanumeric Mode", and a "Graphics Mode".

When used by television sets to present Teletext, the 505x is (normally) used
natively, such that it presents a 7-bit {0x00..0x7F} character set.
	96 Alphanumeric Glyphs - applicable to the language
	64 Graphic Glyphs      - common to all chips
	32 Control Characters  - common to all chips

,----------+ +------------------------------------------------------------.
|    Code: | | 0x00 .. 0x1F  | 0x20 .. 0x3F | 0x40 .. 0x5F | 0x60 .. 0x7F |
|==========| |===============|==============|==============|==============|
|   Alpha: | |               |  Low:Alpha   |              | High:Alpha   |
|~~ Mode ~~+ + Control Codes +~~~~~~~~~~~~~~+~ Mid:Alpha  ~+~~~~~~~~~~~~~~|
| Graphic: | |               |  Low:Graphic |              | High:Graphic |
`----------+ +------------------------------------------------------------'

Each Printable (Alphanumeric/Graphic) Glyph is formed on a 6x10 matrix
	Alphanumeric Glyphs are formed on a 5x9 matrix
		in the TOP-RIGHT corner of the 6x10 matrix,
		leaving ONE pixel to the LEFT, and BOTTOM - for Fixed Kerning & Leading.
		Non-Descending Alphanumeric Glyphs have THREE empty rows on the BOTTOM.
	[qv. GLYPH MAPS]

	Graphical Glyphs may be displayed in either
		# Contiguous Mode
		# Separated  Mode
	[qv. GRAPHICAL GLYPH DESIGN]


 ,------------------.
(  LANGUAGE SUPPORT  )
 `------------------'

There are a total of EIGHT variants of the 505x chip, each characterised by
a different character set targetting the specific language:
	5050 - English (UK) - As used by the Acorn BBC Micro
	5051 - German
	5052 - Swedish
	5053 - Italian
	5054 - Belgian
	5055 - US ASCII
	5056 - Hebrew
	5057 - Cyrillic

Control Codes       are common to ALL 505x chips.
Graphical Glyphs    are common to ALL 505x chips.
Alphanumeric Glyphs   vary between    505x chips.
	[qv. GLYPH MAPS]


 ,----------------.
(  HARDWARE RESET  )
 `----------------'

At Reset, the 505x will default to:
	# TV Picture  : ENabled
	# Concealed   : ENabled
	# Superimpose : DISabled

These features are enabled/disabled by the TV hardware
...classically, by the user by means of a (Infra Red) Remote Control Unit.

Conceal is used to hide Glyphs
...Eg. The answers to quiz questions.

Superimpose sets the background colour to Transparent.
...This was used to allow the user to view the TV Image while reading
Contrary to instinct, this is NOT used to display Subtitles
...that is achieved by use of the "Box-Start/End" Control Codes.


 ,---------------.
(  SCREEN LAYOUT  )
 `---------------'

TEXT RESOLUTION
===============

	Each of the "24+1" [TWENTY-FOUR PLUS ONE] lines of text {0..24}
	has 40 [FORTY] character cells.

	"24+1" (rather than "25") is commonly used to describe a VDU text resolution
	as the bottom row of a VDU was historically reserved-for, and referred-to
	as the "Status Line".

	Broadcast Teletext pages:
		GENERALLY only use 23 of the 25 available lines.
			Line 0  is GENERALLY reserved for {Page No., Service Name, Time, ...}
			Line 24 is GENERALLY reserved for "Fastext" (Colour Button) links.
		COMMONLY only use 39 of the 40 character-slots-per-line for Glyphs.
			Column 0 is COMMONLY reserved for the initial Colour Control Code.

	This document uses Text Coordinates in the form:
		The Top-Left     corner is { 0, 0}
		The Bottom-Right corner is {24,39}  ... Ie. {y, x}

	To ensure the image did not bleed in to the Overscan area of a CRT,
	a space/border equivalent to 6 [SIX] Glyphs was left at both the
	start and end of each line.


			|<6>|<---------- 40 ---------->|<6>|

			+---+--------------------------+---+
			|   |_P100___Teletext____14:30_|   | <-- y=0  : Title
			|   | .........................|   | <--,
			|   | .........................|   |     '
			|   | .........................|   | :    '
			|   | .........................|   |       )-- y=1..23 : Body
			|   | .........................|   | :    ,
			|   | .........................|   |     ,
			|   |__________________________|   | <--'
			|   | INDEX  SPORT  NEWS  HELP |   | <-- y=24 : Status/Fastext
			+---+--------------------------+---+
			     ^-- x=0                  ^-- x=39

	The screen is filled one line at a time, starting at {0,0}

** NOTE **
==========

	The Status line was (generally) NOT included in the Teletext Image,
	but actually programmed by the Fastext feature.

	Also, the 6 Left/Right spaces are outside user control.

	Therefore, in this document, future references to the screen size will
	assume TWENTY-FOUR lines {0..23}, each of FORTY cells {0..39}.

SCREEN DATA SIZE
================

	On the 505x, There are NO Control Codes to position the cursor.
	Therefore EVERY Line   REQUIRES exactly 40 Glyphs/Control-Codes
	      and EVERY Screen REQUIRES exactly 24 Lines

	Making a screen FULL of data be exactly (24 * 40 = 960) bytes

PIXEL RESOLUTION
================

	Each Glyph is exactly {6x10} Pixels {WxH}
	...This includes Alphanumeric Kerning and Leading.

	Therefore the Display has a PIXEL Resolution of:
		{(40*6)            x (24*10)    }  ==  {240 x 240}

RASTER RESOLUTION
=================

	Pixels are GENERALLY rendered with an aspect ration of 1.33:1

	Therefore the Display has an effective RASTER Resolution of:
		{((40*6)*1.33)     x (24*10)    }  ==  {320 x 240}

SCREEN RESOLUTION
=================

	Glyph Pixel rendering is GENERALLY performed as {2x2} CRT Pixels

	Therefore the Display has a SCREEN Resolution of:
		{(((40*6)*1.33)*2) x ((24*10)*2)}  ==  {640 x 480}

	Furthermore, rendering each Glyph Pixel as a {2x2} CRT Pixel allows
	the rendering of "Half Pixels" for use in 505x Font Smoothing algorithm.
		[qv. FONT SMOOOTHING]

	In the analogue/Vector world, Pixel Scaling is achieved by clock skewing.
	In the digital /Raster world, this must be done prior to rendering.
		[qv GLYPH SCALING]


MODE-7 CURSOR POSITIONING
=========================

	Mode-7 adds SEVEN Cursor Positioning Control Codes:
		{Up, Down, Left, Right, Start-of-Line, Start-of-Screen, Go-To}

	All these codes exist in the Mode-7 Control Code Range {0x00..0x1F}

	This is why Mode-7 moves the 505x Control Codes are moved to (0x80..0x9F}


 ,----------------.
(  COLOUR BUTTONS  )
 `----------------'

In Teletext, line #24 (the 25th/Bottom/Status Line) was GENERALLY used for the
Colour-Buttons {Red, Green, Yellow, Cyan/Blue}.

Colour Buttons were (literally) coloured buttons on the TV Remote Control Unit
which would allow the user to quickly access (up to) FOUR chosen pages.

They were displayed and programmed by Teletext Packets 24 & 27,
transmitted on Line 18 in the "Front Porch" of the "Vertical Blanking Interval".

The Analogue Television Transmission Standard is beyond the scope of this document.
This seems to be a good source of information:
	http://www.exampointers.com/teletext/
	https://web.archive.org/web/20220128075223/http://www.exampointers.com/teletext/


 ,----------------.
(  LINE OPERATION  )
 `----------------'

At the start of EVERY display line, the 505x State Machine is reset to:
	Background: Black
	Text/Alpha: White ("alpha" as in "alphanumerics", NOT 'transparency')
	Flash     : Off         !on
	Height    : Normal      !double
	Graphics  : Contiguous  !spaced
	Graphics  : Release     !hold
	Box       : Off         !on
...Notably, EVERY line starts in Alphanumeric Mode

EVERY Code sent to the VDU occupies exactly ONE character cell.
...This INcludes ALL Control Codes (mapped or otherwise).


 ,-------------.
(  PIXEL RATIO  )
 `-------------'

Teletext COMMONLY renders Pixels with a Pixel Ratio of 1.33:1
to achieve a 4:3 IMAGE aspect ratio suitable for a 'standard' 4:3 CRT.

The 1.33:1 Pixel scaling is/was, in the analogue world, performed by clock skewing.

Hardware implementations are beyond the scope of this document.
More information can be found in the 5050/505x datasheet(s).


 ,----------------.
(  FONT SMOOTHING  )
 `----------------'

...I've yet to properly understand this
...it's in the datasheet ...but it's poorly described :/

Essentially "Half Pixels" are added to the empty Pixel space on Glyph diagonals
by analysing adjacent Glyph rows.

The 505x provides TWO (different) algorithms for:
	Single Height Glyphs
	Double Height Glyphs


 ,-------.
(  BOXES  )
 `-------'

Commonly used for subtitles, "NewsFlash" alerts, "ALARM!", etc.
Boxes allow you to specify regions of the Screen which will be rendered.

Unlike Conceal Mode, Boxes imply the use of Blend/Mix/Subtitle mode,
and effects BOTH Glyph rendering AND Background colour.

To begin a region, you must send TWO Box-Start Control Codes (no idea why)
	The first  will be invisible,
	The second will render (as a space, by default).

In Subtitle Mode, only areas of each line prefixed with Box-Start are rendered.
Rendering may be stopped with TWO Box-End Controls (again, no idea why 'two')
	The first  will render (as a space, by default).
	The second will be invisible,

If no Box-End is found on the line,
...Box Mode will be disabled automatically at the end of the line

You MAY set the Background colour BEFORE the first Box-Start
...So you CAN control the COLOUR of the first charcater

You MAY "Hold" a Graphic Glyph for use as the first character.


 ,-----------------.
(  FLASHING GLYPHS  )
 `-----------------'

Both Alphanumeric and Graphic Glyphs may be set to Flash

The flash ratio is 3:1 (ON:OFF)

The 505x datasheet suggests that should be driven by a 0.75Hz clock
...which would give: 3.99:1.33 seconds
...I don't remember it being this slow :/

Flashing does NOT swap the Background and Foreground colours.
The Background will remain constant,
  and the Glyph will only be rendered during the ON phase.

Any number of Flash-On and Flash-Off Control Codes may appear on any line.
	Flash On           :          FUTURE Glyphs   WILL   Flash
	Flash Off / Steady : THIS and FUTURE Glyphs will NOT Flash

Note that Flash-On and Flash-Off Control Codes MAY render as Graphical Glyphs
IF Graphics-Hold is enabled.


 ,--------------.
(  CONCEAL MODE  )
 `--------------'

The 505x has the ability to Conceal (hide) Glyphs.
These Glyphs can be Revealed by the User (often with a TV Remote Control Unit).

Typically, you might "Conceal" the answer to a question.
The user can press [Show|Hide] (on the TV RCU) to Reveal Concealed Glyphs.
Typically, a second press of [Show|Hide] button re-enables Conceal mode.

Simply, for any given Glyph:
	if ((glyphMode == Conceal) && (userMode == Hide))
		Do NOT render the Glyph.

...or inversely:
	if ((glyphMode == Reveal ) || (userMode == Show))
		Render the Glyph.

As with Flash, Conceal Mode does NOT conceal the Background Colour,
it simply fails to render the Glyph.


 ,---------------.
(  DOUBLE HEIGHT  )
 `---------------'

Double height text is exactly that, a Y scaling of 2

Every Glyph Pixel is rendered double-height

Double Height Glyphs take up exactly TWO lines of the screen

The FIRST  line  {n}  MAY contain BOTH Single and Double Height Glyphs.
The SECOND line {n+1} MAY contain ONLY            Double Height Glyphs.

Mode-7 DOES/MAY allow Single Height Glyphs on the SECOND {n+1} line.
	[depending on what documentation you read]

BOTH lines MUST contain the same Glyphs,
and MUST BOTH contain the Double-Height Control Code.

The Specification simply says
	"...you must arrange for two consecutive display rows to contain identical data"

 ,------------------------.
(  GLYPH COLOUR SELECTION  )
 `------------------------'

ANY Alpha-Colour Control Code sets the:
	# Glyph Colour
    # 505x State Machine to Alphanumeric Mode

ANY Graphic-Colour Control Code sets the:
	# Glyph Colour
    # 505x State Machine to Graphics Mode


 ,-----------------------------.
(  BACKGROUND COLOUR SELECTION  )
 `-----------------------------'

The Background Colour is, by reset to Black at the Start of each line.

Changing the background colour requires TWO (or, arguably THREE) Control Codes:
Each of these codes DOES occupy a Character Cell on the VDU.

	1. Alpha/Graphic Colour (of desired Background Colour) [*Eg. Red]
	2. New Background
	   ...THIS *and* FUTURE Glyphs will be displayed as *Red-on-Red
	3. Alpha/Graphic Colour (of desired Ink Colour)        [*Eg. Yellow]
	   ...THIS *and* FUTURE Glyphs will be displayed as *Yellow-on-Red

The Background may be reverted to Black with a single Control Code

	1. Background: Black


 ,---------------.
(  GRAPHICS MODE  )
 `---------------'

When the 505x is in Graphics Mode,
  the Graphics-Contiguous and Graphics-Spaced
  and Graphics-Hold       and Graphics-Release Control Codes are enabled.
These FOUR Control Codes are commonly, and often cleverly, used in Teletext Art.

There may be any number of Contiguous and Spaced Codes on any line.
	Reading from left-to-right along a single line:
		Contiguous: FUTURE Sixels will adjoin
		Space     : FUTURE Sixels will be spaced
	[qv. GRAPHICAL GLYPH DESIGN]

There may be any number of Hold and Release Codes on any line.
	Reading from left-to-right along a single line:
		Release   : FUTURE Control Codes will map to to SPACE {0x20}.
		Hold      : FUTURE Control Codes will map to
		                       the most recent Graphics Glyph used on this line.

	Hold DOES maintain the current Contiguous/Space Mode.

	NB. ANY change to the Colour or Height of the glyphs WILL act as a Release!

	Notably, in a "Hold...Release" scenario, the Release Code WILL map to
	                           the most recent Graphics Glyph used on this line.


 ,---------------.
(  GLYPH SCALING  )
 `---------------'

For accurate rendering, the Glyphs should be scaled to match the Pixel Ratio.

The 1.33:1 Pixel scaling is, in the analogue world, performed by clock skewing.
...As this is not available to us in the digital world, I proffer these ideas:

THREE-BY-THREE
==============

	Scaling for 3x3 is trivial and perfect.

	Simply render each 1x1 Glyph Pixel as {1.33*3, 1*3} == {4 x 3} Screen Pixels.

	For a full Teletext render you will need a screen resolution of:
		{(((40*6)*1.33)*3) x ((24*10)*3)}  ==  {960 x 720}  {WxH}

TWO-BY-TWO
==========

	The 505x renders each 1x1 Glyph Pixel as a 2x2 Screen Pixels.
	...So this is a very realistic choice

	For a full Teletext render you will need a screen resolution of:
		{(((40*6)*1.33)*2) x ((24*10)*2)}  ==  {640 x 480}  {WxH}

	This idea also effectively performs a level of Alphanumeric Glyph Smoothing.

(  MY TOY  )
 `--------'

The toy I am making uses a (rather nice) 4:3 {320x240} IPS screen.
Namely an Adafruit: 2.0" 320x240 Color IPS TFT Display
	https://www.adafruit.com/product/4311

Which fits PERFECTLY for a 24 row Teletext display with a 1.33:1 Pixel ratio.

However it is only 2.0" (50.8mm) in diameter giving me (approximately)
	{40 x 30}mm of screen estate {WxH}
...That would make each Glyph {1.25 x 1.0}mm ...which is ridiculously tiny :(

So I have the choice:
	# Render classic Teletext pages in a "ridiculously tiny" font
	# DOUBLE the size of every pixel but:
	    Be limited to 12 lines x 20 characters
		Characters will scale NICELY
		Glyph Smoothing can be AUTHENTIC [qv. 505x]
		Text SHOULD be readable
	# TRIPLE the size of every pixel but:
	    Be limited to 8 lines x 13 characters
		Characters will scale PERFECTLY
		Glyph Smoothing will be APPROXIMATE
		Text should be EASILY readable

NB. If you sextuple (6x6) the size of each pixel,
    you can have perfect scaling AND authentic smoothing.
	But you will need a screen of resolution of:
		{(((40*6)*1.33)*6) x ((24*10)*6)}  ==  {1920 x 1440}  {WxH}

(  CONTROL CODE TABLES  )
 `---------------------'

Control Codes are the same in ALL variants of the 505x,
and work in BOTH Alpha Mode AND Graphic Mode.

Although the Control Codes are all 7-bit {0..6} values in the range {0x00..0x1F}
They are COMMONLY referred to as if bit-7 (2^7) were Set, ie. (0x80..0x9F}

The BBC did not handle Hex nicely, so BBC codes are also given in Decimal

 ,------------------------.
(  GRAPHICAL GLYPH DESIGN  )
 `------------------------'

The Graphical Glyphs are formed from SIX (oversized) pixels known as "Sixels"

The top and bottom row   of Sixels are THREE pixels heigh
The    middle      row   of Sixels are FOUR  pixels heigh
Both             columns of Sixels are THREE pixels wide

For artistic reasons [ASCII art has its limitations], in this section,
a SINGLE screen pixel is shown as TWO characters wide {'  ', 'xx', '##'}


               |  +1 |  +2  | <----+--- little-endian
								   |
    ___  ___   +-----|------+      V
     ^    ^    |      ##xx##|          _
     |    3    |      xx##xx|  = "--2"  `.     Digits are effectively base-4
     |   _v_  _|______##xx##|_            \           Ie. {0..3} [2 bits/digit]
     |    ^    |##xx##|     |              \
     |    4    |xx##xx|     |  = "-1-"      > "012"  <-- Sixel name
    10    |    |##xx##|     |              /   ^^^
     |   _v_  _|xx##xx|_____|_            /    |||
     |    ^    |      |     |            ,     ||`--- Top row [last]
     |    3    |      |     |  = "0--" _.      |`---- Middle row
    _V_  _v_   |      |     |                  `----- Bottom row [first]
               +------|-----+

    \          |<-3-> |<-3->|
     Pixels
           \_  |<----6----->|


This numbering system implies that the Sixels exist in the character set
in increasing numerical order.

There is also a "Separated Graphics Mode", in which each Sixel is missing
ONE pixel from the LEFT and BOTTOM.

"Separated Graphics Mode" does NOT imply a second set of Graphical Glyphs,
but a different way to render the Glyph data.
...But implement/simulate it however you want <shrug>

Here {'..', ',,'} is used to highlight which pixels are rendered as 'unplotted'.

               |  +1 |  +2  | <----+--- little-endian
								   |
    ___  ___   +-----|------+      V
     ^    ^    |..    ,,xx##|          _
     |    3    |..    ,,##xx|  = "--2"  `.     Digits are effectively base-4
     |   _v_  _|..,,..,,..,,|_            \           Ie. {0..3} [2 bits/digit]
     |    ^    |..xx##,,    |              \
     |    4    |..##xx,,    |  = "-1-"      > "312"  <-- Sixel name
    10    |    |..xx##,,    |              /   ^^^
     |   _v_  _|..,,..,,....|_            /    |||
     |    ^    |..xx##,,xx##|            ,     ||`--- Top row [last]
     |    3    |..xx##,,xx##|  = "3--" _.      |`---- Middle row
    _V_  _v_   |..,,..,,..,,|                  `----- Bottom row [first]
               +------|-----+

    \          |<-3-> |<-3->|
     Pixels
           \_  |<----6----->|

#!/bin/bash

##+============================================================================
doit () {
	file=$1
	hdr=$2

	## Grab the "XML Tags"
	tag="`grep "^<glyphMap id=$hdr " $file`"
	mode=`echo "$tag" | sed 's/.* mode=\([^ ]*\)[ \>].*/\1/'`
	cnt=`echo "$tag" | sed 's/.* cnt=\([^ ]*\)[ \>].*/\1/'`
	chr0=`echo "$tag" | sed 's/.* chr0=\([^ ]*\)[ \>].*/\1/'`

	## We need to know this so we don't put a comma at the end of the array
	((last = cnt -1))

	## Start the array for this Glyph Set
	echo -e "// Teletext Glyph Map: $hdr"
	echo "uint8_t  $hdr[$cnt][10] = {  // Mode: $mode,  Base offset: $chr0 "

	## Parse the glyphs
	idx=0                                                  ## Array index
	n=0                                                    ## Rows counter

	while read -r l ; do                                   ## Read a line
		[[ -z $l ]] && continue                            ##   Ignore empty lines

		eval "bits${n}=(\$l)"                              ##   Get a full line of 16 rows - to an array
		((++n < 10)) && continue                           ##   ...until we have 10 lines

		for ((gly = 0;  gly < 16;  gly++)) ; do            ##   for 16 glyphs per line

			printf -v chrx "%02X" $((chr = idx + chr0))    ##     Find chr$() values
			eval "asc=\x$chrx"                             ##     Find ASCII character for value

			## Start array for this Glyph
			echo -en "\t{ // [$idx] CHR\$($chr) '\\\\x$chrx'"
			if ((chr >= 32 && chr < 127)) ; then
				echo -en " '\\$asc' ----------------------------------------\n\t"
			else
				echo -en " --------------------------------------------\n\t"
			fi

			## Generate the Glyph array element data
			for ((row = 0;  row < 10;  row++)) ; do        ##     for 10 rows
				eval "x=\${bits$row[$gly]}"                ##       Get a single row of bits

				val=0                                      ##       Calculate value
				for ((bit = 0;  bit < 6;  bit++)); do      ##       6 bits per row
					((val <<= 1))                          ##         Make way for new bit
					[[ ${x:bit:1} != . ]] && ((val |= 1))  ##         Add bit in (if present)
				done                                       ##       .
				printf -v valx "0x%02X" $val               ##       Convert to hex

				if ((row == 10-1)); then                   ##       Last row of glyph?
					echo -en "  $valx  /* $x */\n\t}"      ##         No comma after value; close array
					((idx != $last)) && echo -e ","        ##         NOT the last glyph? .. Append a comma
				else                                       ##       Glyph row 0..8
					echo -en "  $valx, /* $x */\n\t"       ##         Comma after value; leave array open
				fi                                         ##       .
			done                                           ##     /next row

			((idx++))                                      ##     Next glyph index
		done                                               ##   /next gly

		n=0
	done < <(sed -n "/^<glyphMap id=$hdr /,/^<\/glyphMap>/p" $file | sed '1d;$d')

	echo -e "\n};"
}

##+============================================================================

# Parse all maps
for g in `grep "^<glyphMap " $src | sed 's/.* id=\([^ ]*\)[ \>].*/\1/'`; do
	doit $src $g
done

exit 0

typedef
	enum ttClr {
		TTC_BLK = 0,
		TTC_RED = 1,
		TTC_GRN = 2,
		TTC_YEL = 3,
		TTC_BLU = 4,
		TTC_MAG = 5,
		TTC_CYN = 6,
		TTC_WHT = 7,
	}
ttClr_t;

typedef
	enum ttBox {
		TTB_OFF,
		TTB_ST1,
		TTB_ST2,
		TTB_ND1,
		TTB_ND2,
	}
ttBox_t;

typedef
	struct fastext {
		char     title[10];
		ttClr_t  clr;
		int      page;
	}
fastext_t;

typedef
	struct teletext {
		struct {  // Hardware parameters
			int        w;        // columns
			int        h;        // rows
			
			int        blkEn;    // Black Available

			bool       fastext;  // Fastext support
			fastext_t  ft[4];    // Fastext buttons (if supported)
		};
		struct {  //  Internal State Machine
			int        y;        // Cursor y coord
			int        x;        // Cursor x coord        **0

			ttClr_t    clrFg;    // Foreground colour     **WHITE
			ttClr_t    clrBg;    // Background colour     **BLK
			bool       flash;    // !steady               **OFF
			bool       conceal;  // !reveal               **OFF

			bool       dblH;     // !single height        **OFF

			bool       gfx;      // !alphanumeric mode    **OFF
			bool       spaced;   // !contiguous graphics  **OFF
			bool       hold;     // !release              **OFF

			ttBox_t    box;      // box status            **OFF
		};
		struct {  // User operations
			bool       show;     // !hide    [un-conceal]
			bool       scroll;   // !paged   [Mode-7]
			bool       subtit;   // subtitle [render only boxes]
			bool       mix;      // subtitle [background transparent]
		};
		struct {  // Character set
			int        m505x;
			glyph_t*   gfx_low;
			glyph_t*   gfx_high;
			glyph_t*   alpha_low;
			glyph_t*   alpha_med;
			glyph_t*   alpha_high;
		};
	}
teletext_t;
