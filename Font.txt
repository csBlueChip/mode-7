(  BBC MODE-7  )
 `------------'

BBC Mode-7 is actually a special use-case of
	Mullard SAA-5050 Teletext Character Generator [505x - English (UK) Variant]

The Acorn BBC Micro has some notable anomalies.
	# Character set is presented as two 256 character fonts
	# A few strange character re-mappings
	# Additional Control Codes


 ,------------------.
(  MULLARD SAA-505x  )
 `------------------'

By current [2022] ways of thinking, the 505x might be thought of as presenting
"two fonts", but it is a /single/ VDU "Character Set", which all (commonly)
have/had an "Alphanumeric Mode", and a "Graphics Mode".

When used by television sets to present Teletext, the 505x is (normally) used
natively, such that it presents a 7-bit {0x00..0x7F} character set.
	96 Alphanumeric Glyphs - applicable to the language
	64 Graphic Glyphs      - common to all chips
	32 Control Characters  - common to all chips

,----------+ +------------------------------------------------------------.
|   Code   | | 0x00 .. 0x1F  | 0x20 .. 0x3F | 0x40 .. 0x5F | 0x60 .. 0x7F |
|==========| |===============|==============|==============|==============|
|  Alpha   | |               |  Low:Alpha   |              | High:Alpha   |
|~~ Mode ~~+ + Control Codes +~~~~~~~~~~~~~~+~ Mid:Alpha  ~+~~~~~~~~~~~~~~|
| Graphic  | |               |  Low:Graphic |              | High:Graphic |
`----------+ +------------------------------------------------------------'

Each Printable (Alphanumeric/Graphic) Glyph is formed on a 6x10 matrix
	Alphanumeric Glyphs are formed on a 5x9 matrix
		in the TOP-RIGHT (^>) corner of the 6x10 matrix,
		leaving ONE pixel to the LEFT (<), and BOTTOM (v) - for Fixed Kerning.
		Non-Descending glyphs have THREE rows of unused pixels on the BOTTOM.
	[qv. GLYPH MAPS]

	Graphical Glyphs may be displayed in either
		# Contiguous Mode
		# Separated  Mode
	[qv. GRAPHICAL GLYPH DESIGN]


 ,------------------.
(  LANGUAGE SUPPORT  )
 `------------------'

There are a total of EIGHT variants of the 505x chip, each characterised by
a different character sets targetted for various languages:
	5050 - English (UK) - As used by the Acorn BBC Micro
	5051 - German
	5052 - Swedish
	5053 - Italian
	5054 - Belgian
	5055 - US ASCII
	5056 - Hebrew
	5057 - Cyrillic

Control Codes       are common to ALL 505x chips.
Graphical Glyphs    are common to ALL 505x chips.
Alphanumeric Glyphs   vary between    505x chips. 
	[qv. GLYPH MAPS]


 ,----------------.
(  HARDWARE RESET  )
 `----------------'

At Reset, the 505x will default to:
	# TV Picture  ENabled
	# Concealed   ENabled
	# Superimpose DISabled

These features are enabled/disabled by the TV hardware
...classically with a Remote Control Unit.

Conceal is used to hide Glyphs
...Eg. The answers to quiz questions.

Superimpose sets the primary background colour to Transparent.
...Eg. To show subtitles.


 ,---------------.
(  SCREEN LAYOUT  )
 `---------------'

Each of the (standard) 24+1 {0,1,..23,24} lines of text has 40 character slots.

Broadcast Teletext pages:
	GENERALLY only use 23 of the 25 available lines.
		Line 0  is reserved for {Page Search, Service Name, Time, ...}
		Line 24 is reserved for FastText/Colour Button functions.
	COMMONLY only use 39 of the 40 available character slots per line.
		Column 0 is COMMONLY reserved for the initial Colour Control Code.

This document uses Text Coordinates:
	The Top-Left     (^<) corner is { 0, 0}
	The Bottom-Right (v>) corner is {24,39}
	

		 ,-- x=0                  ,-- x=39
		 V                        V
		+--------------------------+
		|_P100___Teletext____14:30_| <-- y=0  : Title
		|                          | <--,
		|                          |     '
		|                          | :    '
		|                          |       )-- y=1..23 : Body
		|                          | :    ,
		|                          |     ,
		|__________________________| <--'
		| INDEX  SPORT  NEWS  HELP | <-- y=24 : Status
		+--------------------------+
		                          ^--- {24,39}

The screen is filled one line at a time, starting at {0,0}

On the 505x, There are NO Control Codes to position the cursor.
Therefore EVERY Line   REQUIRES exactly  40  Codes
      and EVERY Screen REQUIRES exactly 24+1 Lines
Making each screen FULL of data be exactly (25 * 40 = 1000) bytes
	  
Mode-7 adds SEVEN Cursor Positioning Control Codes:
	{Up, Down, Left, Right, Start-of-Line, Start-of-Screen, Go-To}
All these codes exist in the Mode-7 Control Code Range {0x00..0x1F}


 ,----------------.
(  LINE OPERATION  )
 `----------------'

At the start of EVERY display line, the 505x State Machine is reset to:
	Background: Black
	Text/Alpha: White ("alpha" as in "alphanumerics", NOT 'transparency')
	Flash     : Off         !on
	Height    : Normal      !double
	Graphics  : Contiguous  !spaced
	Graphics  : Release     !hold
...Notably, EVERY line starts in Alphanumeric Mode

EVERY Code sent to the VDU occupies exactly ONE character cell.
...This INcludes ALL Control Codes (mapped or otherwise).


 ,-------------.
(  PIXEL RATIO  )
 `-------------'

Assuming 24 lines of text (eschewing the "+1" Status Line),
each 40 characters long, Teletext COMMONLY renders Pixels with
a Pixel Ratio of 1.33:1 .. {(40*6*1.33 = 320) x (24*10 = 240)}  {WxH}
to achieve a 4:3 IMAGE aspect ratio suitable for a 'standard' 4:3 CRT.

Furthermore each Glyph Pixel is COMMONLY rendered as a 2x2 Screen Pixel.
This allows the rendering of Half Pixels for use in Font Smoothing.
[qv. FONT SMOOOTHING]

 ,-------.
(  BOXES  )
 `-------'

Commonly used for subtitles, 
Boxes allow you to specify regions of the Screen which will be rendered.
This effects BOTH Glyph rendering AND Background colour.

To begin a region, you must send TWO Box-Start Control Codes
The first  will be invisible,
The second will render (as a space, by default).

In Subtitle Mode, only areas of each line prefixed with Box-Start are rendered.
Rendering may be stopped with TWO Box-End Controls
The first  will render (as a space, by default).
The second will be invisible,

If no Box-End is found on the line, 
...Box Mode will be disabled automatically at the end of the line

You MAY set the Background colour BEFORE you Box-Start
...So you CAN control the COLOUR of the first charcater

You MAY "Hold" a Graphic Glyph for use as the first character.


 ,----------------.
(  FLASH HARDWARE  )
 `----------------'

Both Alphanumeric and Graphic Glyphs may be set to Flash

The flash ratio is 3:1 (ON:OFF)

The 505x datasheet suggests that should be driven by a 0.75Hz clock
...which would give: 3.99:1.33 seconds ...Yeah, probably not :/

Flashing does NOT swap the Background and Foreground colours.
The Background will remain constant, 
  and the Glyph will only be rendered during the ON phase.
  

 ,--------------.
(  CONCEAL MODE  )
 `--------------'

The 505x has the ability to Conceal (hide) Glyphs.
These Glyphs can be Revealed by the User (often with a TV Remote Control Unit).

Typically, you might Conceal the answers to a question.
The user can press [Show|Hide] to disable Conceal mode and check their guess. 
Normally a second press of [Show|Hide] re-enables Conceal mode.

Simply, for any given Glyph:
	if ((glyphMode == Conceal) && (userMode == Hide)) 
		Do NOT render the Glyph.

...or inversely:
	if ((glyphMode == Reveal) || (userMode == Show)) 
		Render the Glyph.

As with Flash, Conceal Mode does NOT conceal the Background Colour,
it simply fails to render the Glyph.

(  DOUBLE HEIGHT  )
 `---------------'

Double height text is exactly that, a Y scaling of 2

Every Glyph Pixel is rendered double-height

Double Height Glyphs take up exactly TWO lines of the screen

The FIRST  line  {n}  MAY contain BOTH Single and Double Height Glyphs.
The SECOND line {n+1} MAY contain ONLY            Double Height Glyphs.

Mode-7 DOES/MAY allow Single Height Glyphs on the SECOND {n+1} line.
	[depending on what documentation you read]

BOTH lines MUST contain the same Glyphs, 
and MUST BOTH contain the Double-Height Control Code.

The Specification simply says

 ,---------------.
(  GRAPHICS MODE  )
 `---------------'

ANY  Alpha  Colour Control Code, sets the 505x to Alphanumeric Mode
ANY Graphic Colour Control Code, sets the 505x to   Graphics   Mode

When the 505x is in Graphics Mode,
  the Graphics-Contiguous and Graphics-Spaced
  and Graphics-Hold       and Graphics-Release Control Codes are enabled.
These Control Codes are commonly, and often cleverly, used in Teletext Art.

There may be any number of Contiguous and Spaced Codes on any line.
	Reading from left-to-right along a single line:
		Contiguous: FUTURE Sixels will adjoin
		Space     : FUTURE Sixels will be spaced
	[qv. GRAPHICAL GLYPH DESIGN]

There may be any number of Hold and Release Codes on any line.
	Reading from left-to-right along a single line:
		Release: FUTURE Control Codes will map to to SPACE {0x20}.
		Hold   : FUTURE Control Codes will map to
		                      the most recent Graphics Glyph used on this line.
	Hold DOES maintain the current Contiguous/Space Mode.
	Most notably, in a "Hold...Release" scenario, the Release Code WILL map to
	                        "the most recent Graphics Glyph used on this line".


 ,---------------.
(  GLYPH SCALING  )
 `---------------'

For accurate rendering, the Glyphs should be scaled to match the Pixel Ratio.

The 1.33:1 Pixel scaling is, in the analogue world, performed by clock skewing.
...As this is not available to us in the digital world, I proffer these ideas:

THREE-BY-THREE
==============

Scaling for 3x3 is trivial and perfect.

Simply render each 1x1 Glyph Pixel as 4x3 Screen Pixels.

For a full Teletext render you will need a screen resolution of:
	40*(6*(3*1.33)) x (24  )*(10*3)  ==  960 x 720  {WxH}
	40*(6*(3*1.33)) x (24+1)*(10*3)  ==  960 x 750

TWO-BY-TWO
==========

The 505x renders each 1x1 Glyph Pixel as a 2x2 Screen Pixels.
...So this is a very realistic choice

For a full Teletext render you will need a screen resolution of:
	40*(6*(2*1.33)) x (24  )*(10*2)  ==  640 x 480  {WxH}
	40*(6*(2*1.33)) x (24+1)*(10*2)  ==  640 x 500

This idea also effectively performs a modicum of Alphanumeric Glyph Smoothing.

 `--------'

The toy I am making uses a (rather nice) 4:3 {320x240} IPS screen.
So assuming an aspect ratio of 1.33:1, and only 24 lines of text, we have a win!

However it is only 2.0" in diameter giving me dimensions of
	{~40 x ~30}mm of screen estate
That would make each Glyph {1.2 x 1.0}mm ...which is ridiculously tiny :(

So I have the choice:
	# Render classic Teletext pages in a "ridiculously tiny" font
	# DOUBLE the size of every pixel but:
	    Be limited to 12 lines x 20 characters
		Characters will scale NICELY
		Glyph Smoothing can be AUTHENTIC [qv. 505x]
		Text will be readable
	# TRIPLE the size of every pixel but:
	    Be limited to 8 lines x 13 characters
		Characters will scale PERFECTLY
		Glyph Smoothing will be APPROXIMATE
		Text will be EASILY readable

NB. If you sextuple (6x6) the size of each pixel, 
    you can have perfect scaling AND authentic smoothing.
	But you will need a screen of resolution:
        40*(6*(6*1.33)) x (24  )*(10*6)  ==  1920 x 1440  {WxH}
        40*(6*(6*1.33)) x (24+1)*(10*6)  ==  1920 x 1500




























 ,-------------------.
(  BACKGROUND COLOUR  )
 `-------------------'

The Background Colour is, by reset to Black at the Start of each line.

Changing the background colour requires TWO (or, arguably THREE) codes:
Each of these codes DOES occupy a Character Cell on the VDU.

	1. Alpha/Graphic Colour (of desired Background Colour) [*Eg. Red]
	2. New Background
	   ...THIS *and* FUTURE Glyphs will be displayed as *Red-on-Red
	3. Alpha/Graphic Colour (of desired Ink Colour)        [*Eg. Yellow]
	   ...THIS *and* FUTURE Glyphs will be displayed as *Yellow-on-Red

 ,------------------------.
(  GRAPHICAL GLYPH DESIGN  )
 `------------------------'

The Graphical Glyphs are formed from SIX (oversized) pixels known as "Sixels"

The top and bottom row   of Sixels are THREE pixels heigh
The    middle      row   of Sixels are FOUR  pixels heigh
Both             columns of Sixels are THREE pixels wide

For artistic reasons [ASCII art has its limitations], in this section,
a SINGLE screen pixel is shown as TWO characters wide {'  ', 'xx', '##'}


               |  +1 |  +2  | <----+--- little-endian
								   |
    ___  ___   +-----|------+      V
     ^    ^    |      ##xx##|          _
     |    3    |      xx##xx|  = "--2"  `.     Digits are effectively base-4
     |   _v_  _|______##xx##|_            \           Ie. {0..3} [2 bits/digit]
     |    ^    |##xx##|     |              \
     |    4    |xx##xx|     |  = "-1-"      > "012"  <-- Sixel name
    10    |    |##xx##|     |              /   ^^^
     |   _v_  _|xx##xx|_____|_            /    |||
     |    ^    |      |     |            ,     ||`--- Top row [last]
     |    3    |      |     |  = "0--" _.      |`---- Middle row
    _V_  _v_   |      |     |                  `----- Bottom row [first]
               +------|-----+

    \          |<-3-> |<-3->|
     Pixels
           \_  |<----6----->|


This numbering system implies that the Sixels exist in the character set
in increasing numerical order.

There is also a "Separated Graphics Mode", in which each Glyph is missing
ONE pixel from the LEFT and BOTTOM of each Sixel.

"Separated Graphics Mode" does NOT imply a second set of Graphical Glyphs,
but a different way to render the Glyph data.

Here {'..', ',,'} is used to highlight which pixels are rendered as 'unplotted'.

               |  +1 |  +2  | <----+--- little-endian
								   |
    ___  ___   +-----|------+      V
     ^    ^    |..    ,,xx##|          _
     |    3    |..    ,,##xx|  = "--2"  `.     Digits are effectively base-4
     |   _v_  _|..,,..,,..,,|_            \           Ie. {0..3} [2 bits/digit]
     |    ^    |..xx##,,    |              \
     |    4    |..##xx,,    |  = "-1-"      > "312"  <-- Sixel name
    10    |    |..xx##,,    |              /   ^^^
     |   _v_  _|..,,..,,....|_            /    |||
     |    ^    |..xx##,,xx##|            ,     ||`--- Top row [last]
     |    3    |..xx##,,xx##|  = "3--" _.      |`---- Middle row
    _V_  _v_   |..,,..,,..,,|                  `----- Bottom row [first]
               +------|-----+

    \          |<-3-> |<-3->|
     Pixels
           \_  |<----6----->|

This code is intended to be readable, and somewhat formatted to reflect the
tables [above].  It is certainly optimisable.

  # Control codes, ASCII-ish
  # Control codes, (mostly) NOT ASCII (exception is #127)
  # Glyphs (mostly ASCII) common to all four modes
  # Special cases
  # Alpha mode    (where lo==hi)
  # Graphics mode (where lo==hi)


// Control codes, ASCII-ish
if (INRANGE(ch, 0x00,0x1F))
{
	switch (ch) {
		case   0 :  return   FN_NOTHING;      //  NUL
		case   1 :  return FN_PRINTER_NEXT;   // (SOH)
		case   2 :  return FN_PRINTER_START;  //  STX
		case   3 :  return FN_PRINTER_STOP;   //  ETX
		case   4 :  return   FN_NOTHING;      // (EOT)
		case   5 :  return   FN_NOTHING;      // (ENQ)

		case   6 :  return FN_VDU_ENABLE;     //  ACK  <----.
		case   7 :  return FN_BEEP;           //  BEL       |

		case   8 :  return FN_CUR_BACK;       //  BS   --.  |
		case   9 :  return FN_CUR_FORWARD;    //  TAB  --|  |
		case  10 :  return FN_CUR_DOWN;       //  LF   --|  |
		case  11 :  return FN_CUR_UP;         // (VT ) --|  |

		case  12 :  return FN_CLS;            //  FF     |  |

		case  13 :  return FN_START_OF_LINE;  //  CR   --'  |

		case  14 :  return FN_MODE_PAGED;     // (SO )      |
		case  15 :  return FN_MODE_SCROLL;    // (SI )      |

		case  16 :  return   FN_NOTHING;      // (DLE)      |
		case  17 :  return   FN_NOTHING;      // (DC1)      |
		case  18 :  return   FN_NOTHING;      // (DC2)      |
		case  19 :  return   FN_NOTHING;      // (DC3)      |
		case  20 :  return   FN_NOTHING;      // (DC4)      |
		case  21 :  return FN_VDU_DISABLE;    //  NAK  <----'

		case  22 :  return FN_MODE_SELECT;    // (SYN)
		case  23 :  return FN_CHR_PROGRAM;    // (ETB)
		case  24 :  return   FN_NOTHING;      // (CAN)
		case  25 :  return   FN_NOTHING;      // (EM )
		case  26 :  return   FN_NOTHING;      // (SUB)
		case  27 :  return   FN_NOTHING;      // (ESC)
		case  28 :  return   FN_NOTHING;      // (FS )
		case  29 :  return   FN_NOTHING;      // (GS )
		case  30 :  return FN_CURSOR_HOME;    // (RS )
		case  31 :  return FN_CURSOR_MOVE;    // (US )
	}
}

// Control codes, (mostly) NOT ASCII (exception is #127)
else if (INRANGE(ch, 0x7F,0x9F))
{
	switch (ch) {
		case 127 :  return FN_DELETE;         //  DEL  [ASCII]
		case 128 :  return   FN_NOTHING;

		case 129 :  return FN_ALPHA_RED;      //
		case 130 :  return FN_ALPHA_GREEN;    //
		case 131 :  return FN_ALPHA_YELLOW;   //
		case 132 :  return FN_ALPHA_BLUE;     //
		case 133 :  return FN_ALPHA_MAGENTA;  //
		case 134 :  return FN_ALPHA_CYAN;     //
		case 135 :  return FN_ALPHA_WHITE;    // *Line Default

		case 136 :  return FN_ATTR_FLASH
		case 137 :  return FN_ATTR_STEADY     // *Line Default

		case 138 :  return   FN_NOTHING;      //+ end box
		case 139 :  return   FN_NOTHING;      //+ start box
		case 140 :  return FN_HEIGHT_NORMAL;  // *Line Default
		case 141 :  return FN_HEIGHT_DOUBLE;
		case 142 :  return   FN_NOTHING;      //+ S0 SO ??
		case 143 :  return   FN_NOTHING;      //+ S1 SI ??
		case 144 :  return   FN_NOTHING;      //+ DLE

		case 145 :  return FN_INK_RED;
		case 146 :  return FN_INK_GREEN;
		case 147 :  return FN_INK_YELLOW;
		case 148 :  return FN_INK_BLUE;
		case 149 :  return FN_INK_MAGENTA;
		case 150 :  return FN_INK_CYAN;
		case 151 :  return FN_INK_WHITE;

		case 152 :  return FN_DISP_CONCEAL;

		case 153 :  return FN_GRAPH_CONTIG;   // *Line Default
		case 154 :  return FN_GRAPH_SPACED;
		case 155 :  return   FN_NOTHING;      //+ ESC

		case 156 :  return FN_PAPER_BLACK;    // *Line Default
		case 157 :  return FN_PAPER_NEW;

		case 158 :  return FN_GRAPH_HOLD;
		case 159 :  return FN_GRAPH_RELEASE;  // *Line Default
	}
}

// Glyphs (mostly ASCII) common to all four modes
else if (INRANGE(ch, 0x40,0x5E) || INRANGE(ch, 0xC0,0xDE))
{
	switch (ch) {
		case 0x5B:  case 0xDB:  return GLYPH_LEFT;
		case 0x5C:  case 0xDC:  return GLYPH_HALF;
		case 0x5D:  case 0xDD:  return GLYPH_RIGHT;
		case 0x5E:  case 0xDE:  return GLYPH_UP;
		// {32, 35, 64, 65 .. 90) == {' ', '#', '@', 'A', ..., 'Z'}
		default:                return GLYPH_ASCII;
	}
}

// Special cases
else if (   ((ch & ~0x80) == 0x23)    // 0x23, 0xA3
         || ((ch & ~0x80) == 0x5F)    // 0x5F, 0xDF
         || ((ch & ~0x80) == 0x60)    // 0x60, 0xE0
         || ((ch & ~0x80) == 0x7F) )  // 0x7F, 0xFF
{
	switch (ch) {
		case 0x23:  return GLYPH_ASCII;  // "#" - hash
		case 0xDF:  return GLYPH_HASH;   // Also "#" - hash
		case 0x7F:  return FN_DELETE;
	}
	if        (mode == ALPHA) {
		switch (ch) {
			case 0xA3:  return GLYPH_GBP;
			case 0x60:  return GLYPH_GBP;
			case 0x5F:  return GLYPH_DASH;
			case 0xE0:  return GLYPH_DASH;
			case 0xFF:  return GLYPH_BLOCK;
		}
	} else if (mode == GRAPHICS) {
		switch (ch) {
			case 0xA3:  return GFX(0,0,3);
			case 0x60:  return GFX(0,0,3);
			case 0x5F:  return GFX(2,0,0);
			case 0xE0:  return GFX(2,0,0);
			case 0xFF:  return GFX(3,3,3);
		}
	}
}

// Alpha mode (where lo==hi)
else if (mode == ALPHA)
{
	switch (ch) {
		case 0x7B:  case 0xFB:  return GLYPH_1QUART;  // 1/4
		case 0x7C:  case 0xFC:  return GLYPH_DPIPE;   // ||
		case 0x7D:  case 0xFD:  return GLYPH_3QUART;  // 3/4
		case 0x7E:  case 0xFE:  return GLYPH_DIVIDE;  // Maths divide symbol
		default                 return GLYPH_ASCII;
	}
}

// Graphics mode (where lo==hi)
else if (mode == GRAPHICS)
{
	int  mask;

	if (INRANGE(ch, 0x20, 0x3F))  mask = 0x1F ;
	else                          mask = 0x3F ;

	int  btm = ((ch & mask) >> 4) & 0x03;
	int  mid = ((ch & mask) >> 2) & 0x03;
	int  top = ((ch & mask)     ) & 0x03;

	return GFX(btm, mid, top);
}

In my opinion, the Alphanumeric Mode makes most sense if the top bit [2^7 = 0x80] is CLEAR [0] ....Ie. {0x00..0x7F}
	'#' is in the correct (ASCII) placement
	'[' is replaced with LEFT arrow
	']' is replaced with RIGHT arrow
	'\' is replaced with HALF [1/2]
	'^' is replaced with UP arrow
	'_' is replaced with DASH (aka HYPHEN) [2 pixels shorter than a '-' MINUS]
	'`' is replaced with GBP "pound" currency symbol
NB. "DOWN arrow" does NOT exist in Mode-7

This script will convert the glyph diagrams in to usable C data.

There are many maanny ways to optimise this output.

------------- >-8 ------------- >-8 ------------- >-8 ------------- >-8 ------------- >-8 ------------- >-8 ------------- >-8 -------------
#!/bin/bash

##================================================================================================================
doit () {
	file=$1
	hdr=$2
	last=$3
	((last--))

	cnt=0  ## Array index
	n=0    ## Line counter

	while read -r l ; do         ## Read a line
		[[ -z $l ]] && continue  ## Ignore empty lines

		eval "bits${n}=(\$l)"      ## Get a full line of bits
		((++n < 10)) && continue   ## ...until we have 10 lines

		for ((col = 0;  col < 16;  col++)) ; do  ##   Max 16 glyphs per line
			[[ -z ${bits0[$col]} ]] && continue  ##     Ignore unused chars

			chr=$(case "$cnt" in                 ##     Find chr value
				("95")  echo 255 ;;              ##       ...
				("96")  echo 163 ;;              ##       ...
				("97")  echo 223 ;;              ##       ...
				("98")  echo 224 ;;              ##       ...
				(*   )  echo $((cnt + 32)) ;;    ##       ...
			esac)                                ##       ...
			printf -v chrx "0x%02X" $chr         ##     Convert to chr value hex

			## Start array for this glyph
			echo -en "\t\t{ // ---------------------[$cnt] -> chr($chr) = $chrx\n\t\t"
			for ((row = 0;  row < 10;  row++)) ; do            ## 10 rows
				eval "x=\${bits$row[$col]}"                    ## Get a row

				val=0                                          ## Calculate value
				for ((bit = 0;  bit < 6;  bit++)); do          ## 6 bits per line
					((val <<= 1))                              ##   Make way for new bit
					[[ "${x:bit:1}" != "." ]] && ((val |= 1))  ##   Add bit in (if present)
				done                                           ## .
				printf -v valx "0x%02X" $val                   ## Get hex value

				if ((row == 9)) ; then                         ## Last row of glyph?
					echo -en "  $valx  /* $x */\n\t\t}"        ##   No comma after value; close array
					((cnt != $last)) && echo -e ","            ##   NOT the last glyph? .. Append a comma
				else                                           ## Glyph row 0..8
					echo -en "  $valx, /* $x */\n\t\t"         ##   Comma after value; leave array open
				fi                                             ## .
			done
			((cnt++))                                          ## Next glyph
		done
		n=0
	done < <(sed -n "/^\*\*+${hdr}.*/,/^\*\*-${hdr}/p" ${file}  | sed '1d;$d')
}

##================================================================================================================
## ENTRY POINT
##
echo "uint8_t  glyph[2][(6 *16) +3][10] = {"

echo -e "\t{ // Alphanumeric Glyphs ==============================================="
doit $0 ALPHA $(((6 *16) +3))
echo -e "\n\t},"

echo -e "\t{ // Graphics Glyphs ==================================================="
doit $0 GRAPHIC $(((6 *16) +3))
echo -e "\n\t}"

echo "};"

exit 0
